# WP-C-07 — Workspace Scope Module 成员管理 endpoints

## Parent

`.scratch/workspace-as-db/PRD.md`

## What to build

在 Workspace Scope Module 内补齐真人成员管理写路径。成员管理不是普通业务 CRUD：它必须原子维护目标 workspace database 内的 `user` 表与 `_system.user_workspace_index`，否则登录 hook、workspace 列表和 switch-workspace 会漂移。

### Endpoints

- `POST /api/workspaces/:slug/members`：工作区管理员预创建普通成员或工作区管理员，输入 email、displayName?、isAdmin。
- `PATCH /api/workspaces/:slug/members/:userId`：修改 human user 的 `is_admin`，并同步 `_system.user_workspace_index.role`。
- `DELETE /api/workspaces/:slug/members/:userId`：软移除成员，写 workspace user 的 `disabled_at` 或等价禁用字段，并写 `_system.user_workspace_index.disabled_at`。

### Rules

- 所有 endpoint 先用 `requireOidc()` 取得调用者 OIDC subject；不要用当前 token 的 `https://surrealdb.com/db` / `https://surrealdb.com/ac` scope 约束目标 workspace，否则用户无法从管理面切换或维护其它可管理 workspace。
- Workspace Scope Module 用 root 读取 `_system.workspace`、`_system.user_workspace_index` 和目标 workspace db 的 human `user` 记录，确认调用者在目标 workspace 中是工作区管理员；不把这个校验散落到前端。
- root 负责原子同写 workspace db `user` 与 `_system.user_workspace_index`。这是 workspace lifecycle / scope 维护路径，不是普通业务 CRUD 代理。
- 前端不直接 INSERT / UPDATE / DELETE human `user` 行；浏览器直连只负责读取花名册和业务数据。
- 架构内无邀请闭环。管理员输入 email 预创建 human user；被预创建用户首次 OIDC 登录时由 access AUTHENTICATE 回填 subject / last_seen_at。
- 移除成员不真删 user record，保留历史归因。

## Acceptance criteria

- [x] admin 调 `POST /api/workspaces/:slug/members` 后，目标 workspace db 有 human `user` 记录，`_system.user_workspace_index` 有对应 active row。
- [x] participant 或非目标 workspace 成员调成员管理 endpoint 返回 403，且两边都不写入。
- [x] role 变更同时更新 workspace db `user.is_admin` 与 `_system.user_workspace_index.role`。
- [x] delete/移除只写禁用字段；历史 office_* / workflow_run /业务记录的 user 引用不悬空。
- [x] `GET /api/session/workspaces` 和 IdP default-scope hook 会过滤 disabled row。
- [x] 失败路径不记录 OIDC token、IdP 管理 token、SurrealDB root credential。

## Agent Brief

**Category:** enhancement
**Summary:** AI 抽屉把 `row-patch-proposal` 暂停事件渲染成逐字段提案卡，用户确认后以当前 workspace session 直连写入已接受字段，并恢复 workflow run。

**Current behavior:**
Router workflow 在行分析子 agent 产出 `RowPatchProposal` 后进入 workflow 暂停态，暂停 payload 经 chat stream 事件推到前端；resume 接口（`POST /api/chat/runs/:runId/resume`，携带 decision）已可用。但前端没有任何渲染或确认路径——提案到达后用户看不到、也无法写入。

**Desired behavior:**
- AI 抽屉收到携带 `row-patch-proposal` 的暂停事件时，渲染提案卡：每个建议字段一行，展示当前值、建议值、依据（basis）与置信度（低置信度有视觉区分）。
- 用户可逐字段"接受 / 忽略"，随后一次"确认写入"：只有被接受的字段，用当前用户的浏览器直连 SurrealDB 会话 UPDATE 到目标记录；写入成功后调 resume（write-confirmed），run 继续至 done。
- "全部忽略"或关闭卡片：调 resume（write-rejected），记录零变更，run 标记取消/结束。
- 写入失败（含 PERMISSIONS 拒绝）：提案卡保留、展示与手工编辑同风格的中文错误提示，**不**以 write-confirmed resume；用户可重试或忽略。
- 字段值按目标字段类型做边界转换：record 引用字段遵守"内存 string、SDK 边界包 RecordId"的仓库规则；datetime 字段转 SDK 的 DateTime 类型。

**Key interfaces:**
- `RowPatchProposal`（shared 的 rpc 类型）——`proposals[]` 含 `field / currentValue / suggestedValue / basis / confidence`，是卡片渲染的输入合约，不新增字段。
- `ChatStreamEvent` 暂停事件——卡片由 stream 事件驱动，不另开拉取通道。
- resume 合约——decision 至少区分 write-confirmed / write-rejected 两类。
- 直连写入复用既有记录更新路径（与编辑器同一套字段约束 / 序列化），不绕开它新写一条 UPDATE 通道。
- 提案卡的接受/忽略/确认状态机抽成纯逻辑层（与 svelte runes 镜像分离），沿用 editor-store 的分层风格。

**Acceptance criteria:**
- [ ] 暂停事件携带 row-patch-proposal 时，抽屉渲染提案卡，逐字段显示当前值 / 建议值 / 依据 / 置信度。
- [ ] 接受部分字段并确认后，仅这些字段被写入数据库（用直连 SELECT 断言），未接受字段保持原值。
- [ ] 写入成功后 run 被 resume 且最终 done；抽屉显示完成态。
- [ ] 全部忽略或关闭卡片后，记录零变更，run 以 rejected 路径结束。
- [ ] 写入失败时卡片保留并显示中文错误，run 不进入 write-confirmed。
- [ ] record / datetime 字段的写入值经过正确的边界类型包装。
- [ ] 纯逻辑层测试：状态机（接受/忽略/确认/失败重试）+ fake 连接的写入断言；不打真 LLM、不依赖真实抽屉 DOM。

# .scratch issue execution path

Updated: 2026-05-21

This repository is in the middle of the web-only pivot. The execution path below is the canonical order for the next agents. The main goal is not to finish every historical idea under `.scratch/`; it is to land the web-only architecture spine first, then reattach AI, resource retrieval, and virtual office on top of that spine.

## Non-negotiable constraints

- Do not revive Electrobun, sidecar windows, external WebView research windows, renderer/main RPC, service JWT, NS-admin, backend business CRUD proxies, or SurrealDB LIVE forwarding.
- Use `pnpm` only.
- New code goes to `server/`, `web/`, and `shared/`; legacy folders are temporary source material only.
- Business data reads/writes default to browser direct SurrealDB. Allowed backend exceptions are Workspace Scope Module, Mastra `/api/chat*`, resource save confirmation SSE, Office dispatcher, and root maintenance paths.
- Mastra implementation work must load the `mastra` skill first.
- SurrealQL implementation work must follow the project SurrealDB rules. If the `surrealdb` skill is unavailable, the agent should state that and verify against existing ADR/schema patterns before writing schema.

## Scheduling rule

Issues marked `needs-triage` are not automatically wrong; many are sufficiently described but still need a final maintainer/agent pass after the previous phase lands. The agent should treat the phase order below as the schedule. Do not pick a downstream issue just because it has enough text if its runtime substrate does not exist yet.

At the time this path was written:

- `ready-for-agent`: `RR-012` only, but implementation should still be queued after the web/server spine exists.
- `needs-info`: `VO-08` only, blocked on forms-module tool exposure.
- Historical `done` resource and agentic-AI issues may contain useful contracts, but pre-pivot execution paths inside them are not current unless a post-pivot note says so.

Recommended order:

1. `WP-C-01` `_system` schema seed
2. `WP-C-02` workspace template SQL
3. `WP-C-03` Workspace Scope Module
4. `WP-C-04` schema migration runner
5. `WP-C-06` workspace create lifecycle
6. `WP-C-07` member management endpoints

Why this order:

- C-03 needs `_system` and workspace template contracts.
- C-04 should exist before broad workspace creation is relied on.
- C-06 proves the full create path.
- C-07 preserves membership/index consistency after create works.
- C-05 is safer after the canonical write paths exist.

Gate:

- `_system.workspace`, `_system.user_workspace_index`, and schema-version tables are idempotently created.
- A new workspace database can be created from `shared/sql/workspace-template`.
- Owner user exists in the new workspace db and `_system.user_workspace_index`.
- `GET /api/session/workspaces`, `POST /api/session/switch-workspace`, and default-scope hook work with fake IdP adapter.
- Member add/role-change/soft-remove updates both workspace db `user` and `_system.user_workspace_index`.
- No permission filtering is duplicated in product queries; permissions live in schema/access.

Parallelism:

- C-01 and C-02 should be coordinated tightly because they define schema conventions.
- After C-03 contracts stabilize, C-04 can proceed while C-06 endpoint wiring starts, but both must share the same template loader.

Run after Phases 1-3 have enough substrate. File movement can start earlier after A, but endpoint/tool-auth work needs B and C.

Recommended order:

1. `WP-D1-01` move Router workflow files and update imports
2. `WP-D1-02` `WorkflowsStorage` in workspace db `workflow_run`
3. `WP-D1-03` tool auth rewrite to caller `surrealSession`

Parallelism:

## Phase 6: Resource Retrieval Web-only V1, RR-012

`RR-012` is spec-ready, but implementation should happen after the server/web spine can host the SSE endpoint and route/panel.

Prerequisite check before starting:

- Server has Hono routes, OIDC middleware, and a way to create caller workspace SurrealDB sessions.
- Web has API/SSE client infrastructure and can render app routes/panels.
- Resource schema/store contracts from the historical resource issues are available in `shared/sql/workspace-template` and server/web code. If not, create a small pre-issue to port the resource schema/contracts before implementing RR-012.

Implement:

- Web route/panel for manual evidence paste.
- Dedicated `POST /api/resources/research/save` SSE action.
- Server-side embedding with provider keys kept on the server.
- Resource item + resource embedding + research session update using caller workspace session.
- No generic resource CRUD proxy.
- No embedding enqueue/retry/reindex endpoint.
- Failed save keeps draft/evidence in the panel and lets user click save again.

Gate:

- SSE emits validation/embedding/persist/session/done progress, and error on failure.
- Fake embedding provider integration test persists resource + embedding and updates `createdResourceIds`.
- Failure test proves no retry/enqueue side effect.
- Frontend state test proves draft/evidence survive failed save.

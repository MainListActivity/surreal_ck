# RR-011 — AI 输入区资源搜索模式切换

## Parent

`.scratch/resource-retrieval/PRD.md`

## Triage context

用户反馈：资源搜索不应该作为 AI 抽屉头部、靠近关闭按钮的独立搜索入口出现。资源搜索本身也是一个子 agent 能力，用户应当能先在输入框里描述要检索的内容，再通过发送按钮上的下拉/切换样式选择“搜索资源”或“发送消息”。

当前实现已经具备 ResourceAgent、`resource-retrieval` router category、资源候选暂停、人工检索 session 和主动补库窗口。这个 follow-up 只调整 AI 抽屉的入口语义和交互位置，不重做资源检索底座。

## Agent Brief

**Category:** enhancement
**Summary:** Move resource search from the AI drawer header into a composer send-button mode switch.

**Current behavior:**
The AI drawer exposes a standalone search/proactive-resource icon in the header next to the close control. Activating it opens a generic research window directly, without using the text the user may have typed into the AI composer as the resource retrieval query.

The composer primary action only sends a general AI message. Resource retrieval can still happen through router classification, but the UI does not give the user an explicit way to say "use this typed description as a resource search."

**Desired behavior:**
The AI drawer header should not place resource search beside the close control. The composer primary action should become a split/dropdown-style action that lets the user switch between at least:

- `发送消息`: keep the existing general AI chat behavior.
- `搜索资源`: submit the current composer text and current AI context as a resource retrieval task handled by ResourceAgent.

The active action mode must be visually clear on the send button itself, using an appropriate icon and label. Switching modes must not clear the typed prompt. In resource-search mode, the prompt is the user's search description; the flow should use the existing resource retrieval behavior: high-confidence answer with citations, middle-confidence resource candidates with multi-select, and miss/low-confidence manual research session using the typed prompt as the query.

**Key interfaces:**

- AI composer action state: should represent the selected action mode (`send message` vs `resource search`) independently from the prompt text.
- AI chat/request contract or router dispatch contract: needs a deterministic way for resource-search mode to route to `resource-retrieval` / ResourceAgent instead of relying only on LLM classification.
- Resource retrieval workflow behavior: existing hit/candidates/miss/manual-research suspend and resume contracts should be reused unchanged.
- Header action area: resource search/proactive resource entry should be removed from the close-control cluster or moved out of that cluster so close remains visually unambiguous.

**Acceptance criteria:**

- [x] The AI drawer header no longer shows a resource search/proactive-resource icon adjacent to the close button.
- [x] The composer primary action supports a dropdown or split-button mode switch with `发送消息` and `搜索资源`.
- [x] The selected mode is visible from the button label/icon without opening the dropdown.
- [x] Switching between modes preserves the current prompt and context tags.
- [x] `发送消息` mode preserves the existing AI chat behavior, history handling, streaming, progress hints, and pending-intent rendering.
- [x] `搜索资源` mode submits the current prompt and current context to ResourceAgent/resource retrieval deterministically, without depending on generic router classification to infer the category.
- [x] Resource-search mode does not open an empty proactive research window first; it starts from the user's typed search description.
- [x] Resource-search mode reuses existing resource retrieval outcomes: citation answer, resource candidate multi-select, continue/manual research, and miss-triggered research session.
- [x] Sending/suspended/disabled states are coherent for both modes, including empty prompt and in-flight request handling.
- [x] Tests cover mode switching, prompt preservation, deterministic resource-routing request behavior, header search removal, and unchanged normal-send behavior.

**Out of scope:**

- Redesigning the research window evidence basket or save-resource flow.
- Changing resource ranking, thresholds, embeddings, or ResourceAgent answer generation semantics.
- Building a full resource library management page.
- Adding legal-case-specific search UI or legal article cards.

- Router categories already include `resource-retrieval`, and the router runtime maps it to `resourceAgent`.
- Resource candidate and manual research suspend/resume flows already exist, so this issue should focus on explicit composer mode routing and UI state.

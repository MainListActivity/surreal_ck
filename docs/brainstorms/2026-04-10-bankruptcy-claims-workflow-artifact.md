# Bankruptcy Claims Workflow Artifact

## Target Team

- 破产管理人团队
- 债权申报受理专员
- 复核律师与项目经理
- 负责外部沟通的助理

## Real Working Path

1. 外部债权人通过公开入口提交债权申报信息、联系人信息和附件。
2. 团队成员在首页先看“最近工作簿”，再进入债权申报总表，不会先进入模板中心或管理后台。
3. 在工作簿里先看表格，再逐条打开详情、链路、最近动态或复核说明，不希望被复杂面板先打断。
4. 对明显缺件、口径冲突、币种不一致、主体名称不一致的记录做补正标记。
5. 每日会反复使用搜索、最近文档、复制分享链接、切换工作簿、查看最近修改。
6. 最终输出仍然依赖一份可信的主表，不能让系统看起来像新的数据工作台。

## Observed Pain Points

- 现有系统把首页做成了“模板/工作台/侧栏组合”，用户无法快速确认该从哪里开始。
- 真实能力已经存在，但入口分散在模板、管理、mock 数据和编辑器扩展里，学习成本过高。
- 错误状态不明确时，用户会把“没有权限”“工作簿被删了”“还在重连”误判成系统坏了。

## Version Drift Points

- 债权人提交后，团队通常会在表格内人工修订字段，导致外部表单与内部主表之间出现口径差。
- 同一债权可能在不同成员手中补充不同备注，最近变更必须可见。
- 当工作簿切换或网络波动时，用户需要确定“看到的是最新受控状态”，而不是空白页。

## Disagreement / Review Points

- 申报金额、币种、计息截止日
- 债权主体和关联主体名称
- 是否已补件、是否进入复核、是否需要驳回
- 公开表单采集字段和内部主表字段是否一一对应

## Tencent Habits Users Rely On

- 首页是左侧导航 + 顶部搜索 + 最近文档列表
- 从首页到编辑器没有视觉断层
- 编辑器第一眼必须是表格，而不是业务面板
- 分享、切换文档、最近修改都应该是轻量动作
- 空状态也要保留同样的页面骨架

## Artifact Answers

- Intake: 使用公开表单单独路由，但文案和入口必须属于同一工作流
- Review: 本期只保留复核入口和结构位置，不实现真实队列
- Number checking: 在主表内完成，辅以详情/复核占位面板
- Disagreement points: 金额、主体、附件、补件是最重要冲突源
- Version drift: 最近动态与受控状态文案必须显式存在
- Daily habits: 搜索、最近文档、打开表格、分享链接、切换工作簿是最高频动作

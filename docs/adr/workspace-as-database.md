# ADR: workspace 映射为独立 SurrealDB database，DDL/DML 走用户会话直接执行

- **Status**: Accepted
- **Date**: 2026-05-16
- **Scope**: `schema/`、`server/`、所有跨 workspace / 跨 db 操作、SurrealDB Access 配置
- **Companions**:
  - [`web-only-pivot.md`](./web-only-pivot.md)（部署形态）
  - [`virtual-office.md`](./virtual-office.md)（业务能力）

## Context

[`web-only-pivot.md`](./web-only-pivot.md) 接受了"Web 前端 + 单容器 Bun server + SurrealDB（同机房内网）"形态。

之前若干稿仍假定**单一 SurrealDB database** 承载全部 workspace 的数据，依赖：

- 在每张表上写复杂的 PERMISSIONS 来跨 workspace 隔离（`workspace.owner = $auth OR workspace IN $auth<-has_workspace_member<-workspace`）。
- 后端持有一个长期 service JWT 代所有用户写动态表。
- 用户的 DDL 操作（建表、加字段）必须由后端 root 权限的 execTemplate 中转。

这套做法在新部署形态下被几个事实持续推翻：

1. **每 workspace 一个 db 是 SurrealDB 自然提供的隔离单元**——一旦走这条路，跨 workspace 隔离不再需要在 PERMISSIONS 里手写，db 边界天然保证。
2. **SurrealDB 的 RECORD access + AUTHENTICATE query 已经能精确表达"哪些用户能登录、能登录后是什么权限"**。把 admin/普通成员/虚拟员工都放在 db 内 user 表里、用字段 + AUTHENTICATE 区分，比"全局表 + 复杂 PERMISSIONS"清楚得多。
3. **DDL 自服务**——既然每个 workspace 是自己的 db，让管理员的 JWT 直接拥有 DDL 能力是最直白的；不再需要 root execTemplate 中转 90% 的 schema 变更。
4. **service JWT 概念不再必要**——所有写都以"用户身份"执行，归因天然在 `$auth`，不需要在表字段里手工标 `from_*`。

## Decision

### 1. 拓扑

```
namespace: main
├── database: _system                     — 极简：仅 2 张表，无 access，仅 root 凭证可访问
│     ├── TABLE workspace                — 全局 workspace 索引
│     │     • db_name (string, unique)    — 该 workspace 的 db 名
│     │     • owner_subject (string)      — 创建者的 OIDC subject
│     │     • slug (string, unique)       — URL 展示用
│     │     • name, created_at
│     └── TABLE user_workspace_index     — (email / OIDC sub) → 可进入的 ws db 倒查表
│           • email (string, indexed)     — 用户加入时即填；登录前唯一定位锚点
│           • subject (option<string>, indexed) — 用户首次 OIDC 登录后由后端补写
│           • workspace (record<workspace>)
│           • role ('admin' | 'participant')  — 缓存身份；ws db 的 user.is_admin 为权威
│           • joined_at
│
│     维护规则：
│       • create_workspace execTemplate 在新建 ws db 时同步写一行 user_workspace_index
│         （owner，含 email + subject + role='admin'）
│       • 管理员调 POST /api/workspaces/:slug/members 添加成员时，后端原子写两处：
│         ws db 的 user 表（email + subject=NONE）+ _system.user_workspace_index
│         （email + subject=NONE + role）
│       • 用户首次 OIDC 登录时，AUTHENTICATE 内回填 ws db.user.subject；
│         后端 endpoint 在 SIGNIN 成功后顺手补写 _system.user_workspace_index.subject
│       • admin 翻转某成员 is_admin → endpoint 同步改 _system 倒查表的 role
│       • _system 写入的失败回滚 / 漂移由后端 reconciler 定时校对
│
└── database: ws_<auto-generated-id>      — 每 workspace 一个 db，db 名由系统自动生成（见 §2.1）
      ├── TABLE user                      — 该 workspace 的所有人（管理员、普通成员、虚拟员工共表）
      │     • email (string, unique)      — 真人定位锚点（管理员添加成员时输入此项）
      │     • subject (option<string>, unique) — 真人首次 OIDC 登录时被 AUTHENTICATE 回填；虚拟员工创建时即填 urn:virtual:<uuid>
      │     • kind (string)               — 'human' | 'virtual'
      │     • is_admin (bool)             — 仅 human 用；区分管理员 / 普通成员
      │     • display_name, avatar
      │     • last_seen_at (option<datetime>) — NONE 表示"从未登录"；每次 SIGNIN 在 AUTHENTICATE 内更新
      │     • virtual_profile (object?)   — 仅 virtual 时填，含 role / supervisor / status
      ├── TABLE workbook / sheet / edge_catalog / dashboard_page / ...
      ├── TABLE office_role / office_task / office_message / office_report / user_notification / employee_credential
      ├── TABLE ent_* / rel_*             — 动态业务表
      │
      │  ── 三条 ACCESS：身份隔离落在 DB 引擎层，不依赖应用层兜底 ──
      │
      │── DEFINE ACCESS admin ON DATABASE TYPE JWT URL '<oidc-jwks>'
      │     AUTHENTICATE {
      │       -- 1. 先按 subject 查（老用户路径）
      │       LET $u = SELECT * FROM user WHERE subject = $token.sub LIMIT 1;
      │       -- 2. 没命中 → 按 email 查（首次登录路径），命中则回填 subject
      │       IF $u = NONE AND $token.email != NONE {
      │         LET $u = SELECT * FROM user WHERE email = $token.email LIMIT 1;
      │         IF $u != NONE AND $u.subject = NONE {
      │           UPDATE $u.id SET subject = $token.sub;
      │         };
      │       };
      │       IF $u = NONE OR $u.kind != 'human' OR $u.is_admin != true {
      │         THROW "not an admin of this workspace";
      │       };
      │       UPDATE $u.id SET last_seen_at = time::now();
      │       RETURN $u;
      │     };
      │     -- JWT access 默认拥有 db owner 级能力（DDL + DML）。AUTHENTICATE 硬卡
      │     -- "user 表里有该真人且 is_admin=true"——管理员必须由其他管理员预先 INSERT
      │     -- 到 user 表（输入 email），不存在自动建用户的路径。
      │
      │── DEFINE ACCESS participant ON DATABASE TYPE RECORD
      │     WITH JWT URL '<oidc-jwks>'
      │     AUTHENTICATE {
      │       LET $u = SELECT * FROM user WHERE subject = $token.sub LIMIT 1;
      │       IF $u = NONE AND $token.email != NONE {
      │         LET $u = SELECT * FROM user WHERE email = $token.email LIMIT 1;
      │         IF $u != NONE AND $u.subject = NONE {
      │           UPDATE $u.id SET subject = $token.sub;
      │         };
      │       };
      │       IF $u = NONE OR $u.kind != 'human' OR $u.is_admin = true {
      │         THROW "not a participant of this workspace";
      │       };
      │       UPDATE $u.id SET last_seen_at = time::now();
      │       RETURN $u;
      │     }
      │     DURATION FOR TOKEN 1h, FOR SESSION NONE;
      │     -- RECORD access 无 DDL。AUTHENTICATE 不再"自动 CREATE user"——
      │     -- 普通成员必须由管理员预先添加（POST /api/workspaces/:slug/members）。
      │     -- IF $u.is_admin = true → THROW 是为了防止管理员误走 participant 通道
      │     -- 而获得"看起来像普通成员实际是管理员"的混乱状态。
      │
      └── DEFINE ACCESS employee ON DATABASE TYPE RECORD
            SIGNUP NONE
            SIGNIN {
              LET $u = SELECT * FROM user
                WHERE subject = $subject AND kind = 'virtual' AND virtual_profile.status = 'active'
                LIMIT 1;
              IF $u = NONE { THROW "no such employee"; };
              LET $c = SELECT VALUE secret FROM employee_credential WHERE employee = $u.id LIMIT 1;
              IF $c != $secret { THROW "bad secret"; };
              RETURN $u;
            }
            DURATION FOR SESSION 1h;
            -- 同 participant：RECORD access 无 DDL。
            -- secret 由 dispatcher 在创建员工时本地缓存（issue 03）；SIGNIN query
            -- 内部读 employee_credential 走 root 视角，不受 PERMISSIONS NONE 限制
            -- （SIGNIN 是定义在 db 级的特权脚本，实测可读 PERMISSIONS NONE 表）。
```

### 2. 用户身份与 access 选择

三类身份在 DB 引擎层做硬隔离。后端在用户 OIDC 验证通过后，按下面这张表选 SIGNIN 走哪条 access：

| 身份 | user 表记录 | 走的 access | DB 引擎能力 | $auth |
|---|---|---|---|---|
| **工作区管理员** | `kind='human', is_admin=true` | `admin` (TYPE JWT) | DDL + DML | user 记录 |
| **普通成员** | `kind='human', is_admin=false` | `participant` (TYPE RECORD) | 仅 DML（DDL 被引擎拒） | user 记录 |
| **虚拟员工** | `kind='virtual'` | `employee` (TYPE RECORD) | 仅 DML（DDL 被引擎拒） | user 记录 |

关键性质：

- **DDL 权限的硬隔离落在 SurrealDB 引擎层**：JWT access 用户默认 db owner 级，RECORD access 用户被引擎硬拒任何 DDL。代码层完全不用兜底校验"该用户能不能 DDL"。
- **三类身份共用一张 user 表**，PERMISSIONS / FK / `$auth` 都不需要 union 不同表。
- **后端登录流程**：用户带 OIDC token 调后端 `POST /api/sessions`；后端 OIDC verify 后用 root 查 `_system.user_workspace_index WHERE subject = $token.sub OR email = $token.email`，按 `role` 字段决定每个 workspace 走 admin 还是 participant SIGNIN，把短期 token 缓存在 session。**无须"先试 admin 再试 participant"逐个探测**——倒查表给了权威答案。
- **首次登录路径**：用户首次登录时倒查只能按 email 命中（subject 字段还是 NONE）。SIGNIN 成功后后端立即把 `subject` 补写到倒查表对应行——下次登录就走 subject 命中（更快、更稳）。
- **同一真人在不同 workspace 中可以是不同身份**：A 工作区是管理员，B 工作区是普通成员。这天然由各 ws db 内的 user.is_admin 决定；倒查表里写两行，role 不同。
- **虚拟员工的 SIGNIN 由 dispatcher 负责**：详见 [`virtual-office.md`](./virtual-office.md) §1。

### 2.2 成员添加（无邀请闭环）

本架构**不提供邀请态**——管理员直接预创建成员的 user 记录：

1. **管理员添加成员**：调 `POST /api/workspaces/:slug/members { email, isAdmin?, displayName? }`。
2. **后端**：
   a. admin SIGNIN 通过校验调用者身份。
   b. admin 会话执行 `INSERT user { email, kind: 'human', is_admin, display_name, subject: NONE, last_seen_at: NONE }`。
   c. root 会话写 `_system.user_workspace_index { subject: NONE (email 暂代), workspace, role }`——subject 在该用户首次 OIDC 登录后由 AUTHENTICATE 回填，**倒查表需要额外的 email 索引或在用户首次登录时由后端补写 subject**（详见 [`workspace-as-database.md`](./workspace-as-database.md) Open Question §5）。
3. **被添加用户首次登录**：用 OIDC 登录后调 `POST /api/sessions { workspaceSlug }`，admin / participant access AUTHENTICATE 中按 `email` 命中 user 记录、回填 `subject`、更新 `last_seen_at`。后续直接按 subject 登录。
4. **从未登录的成员**：`user.last_seen_at = NONE`；UI 在花名册中显示"从未登录"。
5. **移除成员**：admin 调 `DELETE /api/workspaces/:slug/members/:id`，admin 会话 DELETE user 记录（user 表的 delete PERMISSIONS 限 admin） + root 删 `_system.user_workspace_index` 对应行。

### 2.1 workspace database 命名

db 名形如 `ws_<id12>`，其中 `<id12>` 由系统自动生成（建议 12 字符 base32 / nanoid，去除易混淆字符）。**用户输入的 slug 仅用于 URL 展示和 _system.workspace.slug 字段**，不进入 db 名。

理由：

- 杜绝 slug 注入 SQL 风险。
- 避免 slug 改名时 db 名跟随（db rename 在 SurrealDB 是一个重操作）。
- 不暴露内部命名给用户。

`_system.workspace.db_name` 记录系统生成的 db 名；前端永远引用 slug，后端自己查映射。

### 3. _system database 的职责

`_system` 只承载两张表：

- `workspace` 索引（slug ↔ db_name，全局唯一）
- `user_workspace_index` 倒查表（email / OIDC sub → 可进入的 ws db）

**不放**到 `_system`：

- 用户主表（用户在每个 workspace db 里独立存）
- 任何业务数据
- 任何邀请态（架构内不存在邀请，管理员直接预创建成员记录）
- schema_version（每个 ws db 自己版本化）

**关键**：`_system` 上**没有 access**——它对真人和虚拟员工完全不可见。只有后端持有 SurrealDB root 凭证才能读写它。这把"系统元数据"与"业务数据"在 SurrealDB 层面就分开了：业务 db 即便整个被攻破，也看不到其他 workspace 的存在。表本身不需要写 PERMISSIONS，因为只有 root 可访问。

### 4. workspace 创建走 execTemplate（root 权限）

新增 `execTemplate` 名 `create_workspace`：

入参（来自后端 endpoint，已校验调用者是合法 OIDC 用户）：

```
{
  ownerSubject: string,
  ownerEmail: string,
  ownerDisplayName?: string,
  workspaceName: string,
  workspaceSlug: string,
}
```

模板执行（root 权限）：

1. 后端先生成 `db_name = 'ws_' + nanoid(12)`，并对 `_system.workspace.db_name` 做唯一索引兜底冲突重试
2. `DEFINE DATABASE <db_name>`
3. `USE DB <db_name>;`
4. `DEFINE ACCESS admin ON DATABASE TYPE JWT URL <oidc-jwks> AUTHENTICATE { ... };`
5. `DEFINE ACCESS participant ON DATABASE TYPE RECORD WITH JWT URL <oidc-jwks> AUTHENTICATE { ... };`
6. `DEFINE ACCESS employee ON DATABASE TYPE RECORD SIGNIN { ... };`
7. seed 全部业务表 schema（含 user / office_role / office_task / office_message / office_report / user_notification / employee_credential / workbook / sheet / edge_catalog / dashboard_page / …）
8. INSERT 第一条 user：`{ email: ownerEmail, subject: ownerSubject, kind: 'human', is_admin: true, display_name: ownerDisplayName, last_seen_at: time::now() }`（创建者通常会立即首次登录，提前填 last_seen_at；若希望严格，可留 NONE 让首次 SIGNIN 时回填）
9. 回到 `_system` db，原子写两行：
   - `INSERT workspace { db_name, owner_subject, name, slug }`
   - `INSERT user_workspace_index { email: ownerEmail, subject: ownerSubject, workspace: <new id>, role: 'admin' }`

模板**只做这一件事**。其余任何动作都不走 execTemplate。所有 _system 写入都只发生在 execTemplate 或后端 reconciler / 成员管理 endpoint 内部，全程 root 凭证。

### 5. 后续所有 DDL/DML 走用户会话

- 用户的 OIDC token 通过后端 endpoint 进来；后端按 §2 的"按身份选 access"流程拿到对应 token，透传查询 / 写入。
- **管理员**：走 `admin` access，token 直接拥有 DDL 能力，直接执行 `DEFINE TABLE` / `DEFINE FIELD` 等。
- **普通成员**：走 `participant` access，token 在 DB 引擎层就被拒任何 DDL；DML 受 PERMISSIONS 约束。
- **虚拟员工**：走 `employee` access，同样无 DDL；DML 受 PERMISSIONS 约束；归因 `$auth = user 记录`。

### 6. 跨 workspace 查询的处理

用户登录后第一步：后端**用 root** 在 `_system.user_workspace_index` 倒查"你能进哪些 workspace + 各是什么 role"，把列表返给前端。用户选一个 workspace 后，前端的所有后续请求带上 `?ws=<slug>`，后端据此决定连哪个 db。

无需 SurrealDB 跨 db SELECT 能力。架构清晰、不依赖特性版本。

### 7. 后端连接管理

- **一条 root 连接到 `_system`**：用于 `user_workspace_index` 倒查 / 写入、`workspace` 索引读写。
- **按需 root 连接到 ws db**：仅 execTemplate / `employee_credential` 写入等少数场景。
- **每个活跃 workspace × user/employee 的 SIGNIN 会话**：dispatcher / API endpoint 在用户请求生命周期内复用，到期靠 secret/oidc-token 续约。
- 虚拟员工执行窗口：dispatcher 用 employee secret SIGNIN 该 ws db → 短期 token 用完即弃。

### 8. service JWT 概念取消

之前几稿中的 `service JWT` 整体不再存在。所有 SurrealDB 写入要么是真人会话、要么是虚拟员工 RECORD access 会话。后端**唯一**长期凭证是 SurrealDB root（仅来自环境变量），用于：

- `create_workspace` execTemplate
- `_system` 读写
- ws db 内极少数 PERMISSIONS NONE 表的写入（如 `employee_credential` 创建，详见 issue 03）

## Consequences

### 正面

- **隔离天然**：跨 workspace 隔离由 db 边界保证，不靠手写 PERMISSIONS。
- **PERMISSIONS 大幅简化**：每张业务表只需要表达"我对这条记录的具体角色"，不需要嵌套 workspace 归属判断。
- **DDL 自服务**：管理员直接改自己 workspace 的 schema，零中转。
- **审计天然**：每条记录的 changefeed `$auth` 字段就是真实操作者，无需在字段里手工归因。
- **service JWT 攻击面消失**：后端无长期凭证（root 凭证仅在 `create_workspace` 内使用，可短期化或借助密钥管理服务）。
- **虚拟员工身份原生**：SurrealDB RECORD access 直接表达，不需要额外签发服务。

### 负面 / 必须正面解决

- **db 数量增长**：SurrealDB 对单 namespace 内 db 数量是否有性能/管理上限需要确认（issue 阶段验证）。预期数千 workspace 量级 OK，更高需要 sharding。
- **schema 变更分发**：业务表 schema 升级时，所有 ws db 都要跑迁移。需要一个迁移 runner（启动时遍历 `_system.workspace` → 逐 db 应用 schema 增量）。
- **跨 db 报表**（如"这个 OIDC 用户全部 workspace 的合计 token 用量"）需要在后端内做 fan-out 查询，不能一句 SQL 搞定。MVP 不做这种报表，长期接受复杂度。
- **root 凭证持有面扩大**：除了 execTemplate，还需要日常用 root 读写 _system 倒查表、写 employee_credential 等。root 凭证从"仅启动时用一次"变成"日常工作连接"，必须独立监控连接审计。
- **一个真人在多个 workspace 中是多条 user 记录**：用户改 display_name / avatar 时要 fan-out 到所有 workspace db。MVP 接受手动同步或在 `_system` 加一个 `user_profile` 缓存表。
- **倒查表漂移风险**：admin 翻转某成员 is_admin 时同步改 _system；若失败需要 reconciler 兜底。MVP 加启动时一次性校对 + 每小时心跳校对。

## Alternatives Considered

### A. 单 db + 复杂 PERMISSIONS（之前几稿）

抛弃。理由：PERMISSIONS 嵌套深、跨 workspace 隔离全靠人工写表达式、service JWT 攻击面大、DDL 必须中转。

### B. 每 workspace 一个 namespace（而不是 db）

抛弃。理由：namespace 在 SurrealDB 中更接近"租户隔离"，但 namespace 之间无法共享任何东西；本架构虽然 `_system` 内东西很少，仍需要 root 跨 db 倒查与认领，namespace 模型让 `_system` 和 ws_* 之间彻底无法跨。db 模型在同 namespace 下更易管理。

### C. 用户表全局存在 _system

抛弃。理由：违背"db 是隔离单元"的核心原则；任何 workspace 内的 PERMISSIONS 想引用 user 都必须跨 db；且用户在不同 workspace 中可能有不同 display_name / role / 偏好，全局表反而造成数据耦合。

### D. 虚拟员工用每员工一把长期 JWT（早期稿）

抛弃。RECORD access + 短期 SIGNIN 替代之，无长期凭证保管负担。

## Open Questions

1. **schema 迁移 runner**：MVP 是启动时遍历 `_system.workspace`，还是后台定时？倾向启动时遍历 + 失败显式上报。
2. **OIDC provider 选择**：当前 ADR 草稿用 `https://o.maplayer.top/t/ck/jwks.json`（实测验证用）；正式上线前需要确认是用自部署 Keycloak 还是商用 IdP，影响 access 字面值与发行方约束。
3. **drop database**（退休 workspace）：MVP 不提供，仅支持"归档"标记；真正 drop 留待数据合规章节正式立项。
4. **同 OIDC subject 在 admin 与 participant access 间的切换**：用户从普通成员被提升为管理员后，已签发的 participant token 仍在 DURATION 内有效。MVP 接受这一延迟；高敏环境可以缩短 DURATION 或加 token blacklist；同时 _system 倒查表 role 字段在后端 endpoint 内立即更新。
5. **倒查表漂移**：admin 翻转成员身份或添加 / 删除成员后，若同步写 _system 失败，倒查表与 ws db.user 不一致。MVP 用 reconciler 校对（启动时一次 + 每小时心跳） + 在 endpoint 内做"先写 ws db 再写 _system、失败回滚"的两阶段语义。
6. **从未登录的过期清理**：管理员添加成员后，被添加人始终不来登录，user 记录会永久挂在那里。MVP 接受；可引入"超 N 个月未登录自动归档"策略。
